# Repository Guidelines

## Project Structure & Module Organization

This repository contains a personal Neovim LTS configuration.

- `init.lua`: main configuration entrypoint. It sets leader keys, bootstraps `lazy.nvim`, defines keymaps, and declares plugins.
- `lazy-lock.json`: plugin lockfile generated by `lazy.nvim`; commit it when plugin versions change intentionally.
- Runtime logs such as `.nvimlog` or `nvim.log` are local artifacts and should not be committed.

There are currently no separate Lua modules, tests, or asset directories. If the config grows, prefer moving plugin specs into `lua/plugins/*.lua` and shared helpers into `lua/config/*.lua`.

## Build, Test, and Development Commands

Use the `nvim-lts` binary for all validation in this repository.

- `which nvim-lts`: confirms the target Neovim installation.
- `nvim-lts --version`: checks the LTS version being configured.
- `nvim-lts --headless -i NONE '+Lazy! sync' +qa`: installs or updates plugins and refreshes the lockfile.
- `nvim-lts --headless -i NONE '+lua require("neo-tree")' +qa`: smoke-tests that Neo-tree loads.
- `nvim-lts`: opens the config interactively for manual keymap and UI checks.

## Coding Style & Naming Conventions

Write Lua with two-space indentation, double quotes for strings, and local helper functions for repeated behavior. Keep configuration tables explicit and close to the plugin they configure. Use descriptive keymap `desc` values because `which-key.nvim` surfaces them in the UI.

Prefer names like `open_neo_tree()` and `neo_tree_width()` for local helpers. Avoid broad refactors when adding a small plugin or keymap.

## Testing Guidelines

There is no automated test suite. Validate changes with headless startup checks and one interactive session. For plugin changes, run `Lazy! sync`, require the plugin module if applicable, and manually confirm keymaps and UI behavior in `nvim-lts`.

## Commit & Pull Request Guidelines

The existing history uses Conventional Commit style, for example `feat: Added init.lua with lazynvim plugin loader & which keys`. Continue using short prefixes such as `feat:`, `fix:`, `chore:`, or `docs:`.

Pull requests should describe the user-facing Neovim behavior changed, list validation commands run, and mention any lockfile updates. Include screenshots only for visual UI changes where they clarify behavior.

## Agent-Specific Instructions

Do not use other Neovim binaries such as `nvim` for validation. Preserve unrelated local artifacts and untracked logs unless explicitly asked to clean them up.
